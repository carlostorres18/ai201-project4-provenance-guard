# Provenance Guard

Provenance Guard is a lightweight Flask API that accepts text submissions and produces an attribution decision (likely human / uncertain / likely AI) plus a transparency label. It also supports appeals and maintains a structured audit log for grading visibility.

## Setup

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Create a `.env` file (never commit it):

```bash
GROQ_API_KEY=...
PORT=8000
```

Run locally:

```bash
python main.py
```

## API

### POST /submit

Request:

```json
{
  "text": "...",
  "creator_id": "test-user-1"
}
```

Response (fields required for later milestones):

- `content_id` (UUID for this submission)
- `attribution` (`likely_human` | `uncertain` | `likely_ai`)
- `confidence` (0.0–1.0)
- `label` (varies by confidence; see below)

### GET /log

Returns the most recent audit log entries:

```bash
curl -s "http://localhost:8000/log?limit=10" | python -m json.tool
```

### POST /appeal

Request:

```json
{
  "content_id": "...",
  "creator_reasoning": "I wrote this myself ..."
}
```

Response:

```json
{
  "content_id": "...",
  "appeal_id": "...",
  "status": "under_review",
  "message": "appeal received"
}
```

## Detection Signals

This project uses two signals because any single heuristic is brittle.

### Signal 1 (Style; Groq-assisted)

- What it measures: a style-based “AI-likeness” score for the text.
- Why this signal: style cues (generic phrasing, hedging, templated transitions) are common in LLM outputs and can be detected even without model logits.
- Output: `style_score` in `[0.0, 1.0]` where `0.0` is human-like and `1.0` is AI-like.

Implementation note: this is a Groq chat call constrained to return structured JSON.

### Signal 2 (Perplexity proxy; local heuristics)

- What it measures: a proxy for predictability (“perplexity-like”) based on surface statistics.
- Why this signal: it gives an independent check that doesn’t rely on LLM judgement.

This signal computes three metrics and combines them:

- Token entropy (lower entropy → more predictable → more AI-like)
- Bigram repetition rate (more repeated bigrams → more AI-like)
- Sentence length standard deviation (lower burstiness → more uniform → more AI-like)

Output: `perplexity_score` in `[0.0, 1.0]` where `0.0` is human-like and `1.0` is AI-like.

## Confidence Scoring (Multi-signal)

Per `planning.md`, both signals are weighted equally:

```text
confidence = (style_score + perplexity_score) / 2
```

The combined confidence is clamped to `[0.0, 1.0]`.

### Thresholds (verified)

These thresholds match `planning.md` exactly:

- `0.00–0.39` → `likely_human`
- `0.40–0.60` → `uncertain`
- `0.61–1.00` → `likely_ai`

Boundary behavior is tested in `test_scoring.py`.

### Example submissions (meaningful variation)

These are real values from local testing (`test_calibration.py`) and show the score is not constant:

- Casual, irregular human anecdote → `confidence = 0.16464248377891078` (`likely_human`)
- Formal academic paragraph (borderline) → `confidence = 0.4159527978731406` (`uncertain`)

If deploying for real, I would replace the heuristic “perplexity proxy” with true token-level perplexity/logprob features from a fixed reference model and add a calibration set (human/AI mixed) to tune thresholds.

## Transparency Labels (exact text)

The label returned by `/submit` changes with the confidence score (it is not constant).

### High-Confidence AI

Title: `AI-Generated (High Confidence)`

Message:

`Our system found strong evidence that this content was generated using artificial intelligence. Confidence: High.`

### High-Confidence Human

Title: `Human-Written (High Confidence)`

Message:

`Our system found strong evidence that this content was written by a human author. Confidence: High.`

### Uncertain

Title: `Attribution Uncertain`

Message:

`Our system found mixed evidence and cannot confidently determine whether this content was written by a human or generated by AI.`

Label threshold behavior is tested in `test_labels.py`.

## Appeals Workflow

- Endpoint: `POST /appeal`
- Effect:
  - Updates the original submission’s `status` to `under_review` in the JSONL audit log
  - Appends a new `event: "appeal"` entry containing `creator_reasoning` and the original decision context
  - Does not automatically reclassify

This matches the architecture narrative in `planning.md` (appeal stored + status updated + audit entry added).

## Audit Log

All submissions and appeals are written to a structured JSONL audit log at `audit_log.jsonl`. Each line is a JSON object (machine-readable), not console output.

### What’s recorded

For `event: "submit"` entries, the log includes:

- `timestamp`
- `content_id`
- `attribution`
- `confidence`
- `style_score` (signal 1)
- `perplexity_score` (signal 2)
- `status` (`classified` or `under_review` if appealed)

For `event: "appeal"` entries, the log includes:

- `timestamp`, `content_id`, `appeal_id`, `creator_reasoning`, `status: "under_review"`
- original decision context: `original_attribution`, `original_confidence`, `original_style_score`, `original_perplexity_score`

## Rate Limiting

This project uses `flask-limiter` to reduce abuse while still supporting realistic usage.

### Configuration

Flask-Limiter 3.x is configured with in-memory storage for local development:

```python
limiter = Limiter(
    get_remote_address,
    app=app,
    default_limits=[],
    storage_uri="memory://",
)
```

### Endpoint limits and reasoning

`POST /submit` is limited to:

- `10 per minute` — supports iterative drafting/testing by a single writer without letting a script flood the system.
- `100 per day` — supports normal coursework/testing volume while preventing sustained automated abuse.

## Known Limitations

- Edited AI content can be misclassified as human: if an AI draft is heavily edited to add personal detail and irregular phrasing, the style signal can drop, and the heuristic perplexity proxy may stay only mid-range. With equal weighting (per spec), the combined confidence can land in `likely_human`.
- Technical / professional writing can look AI-like: formal tone and domain jargon can resemble LLM boilerplate. This mainly affects the style signal (it is sensitive to generic hedging and templated transitions).
- The “perplexity” signal is a proxy, not true perplexity: it uses surface statistics (entropy/repetition/burstiness) and does not compute token-level likelihoods.

## Spec Reflection

- How the spec helped: `planning.md` forced a clear contract for confidence scoring and thresholds, which made it straightforward to implement tests (`test_scoring.py`) that prevent silent drift.
- How the implementation diverged: the spec describes perplexity analysis; in this implementation it’s a local “perplexity proxy” rather than true LM perplexity. This choice keeps the system lightweight and free of additional model dependencies, but it would be replaced in a production deployment.

- Scaffold + integration: I used an AI assistant to scaffold the Flask app structure and initial `/submit` route, then revised the request/response schema to match the project contract (`text`, `creator_id`, and `content_id` for appeal linkage).
- Calibration + debugging: I used an AI assistant to integrate the Groq client and structured JSON outputs, then overrode defaults after discovering a decommissioned Groq model and added explicit scoring tests (`test_scoring.py`, `test_labels.py`) to keep behavior aligned with `planning.md`.
